01-infrastructure.mmd

Deployment & Infrastructure Topology

North-South: Client → Nginx → API Gateway → Service. East-West: internal Docker DNS, không qua Gateway. Tất cả 4 business services pull config từ config-server và register vào eureka-server. PostgreSQL là 1 instance với 4 database riêng. Observability stack (Prometheus/Loki/Tempo/Grafana) nhận metrics, logs, traces từ toàn bộ services qua Micrometer/OpenTelemetry.

03-redemption-flow.mmd

Redemption Flow — Choreography SAGA (happy path)

Toàn bộ giao tiếp giữa services đi qua RabbitMQ với Transactional Outbox — mỗi mũi tên publish thực chất là: INSERT outbox trong transaction chính, sau đó @Scheduled relay (mỗi 5s, FOR UPDATE SKIP LOCKED) mới publish thật lên broker. 
sequenceDiagram
    autonumber
    participant U as Client
    participant O as order-service
    participant MQ as RabbitMQ
    participant C as coupon-service
    participant W as Wallet Mock

    U->>O: POST /orders
    O->>O: INSERT orders + outbox(ORDER_PLACED)
    O-->>U: 202 Accepted (txn_id)
    O->>MQ: relay publish: order.placed
    MQ->>C: order.queue
    C->>C: Redis lock user_lock:{user_id} (TTL 60s)
    C->>C: SELECT coupons ... FOR UPDATE
    C->>C: matching logic (TODO)
    C->>C: INSERT transaction_tracking (INIT)
    C->>C: INSERT outbox(REDEEM_COMMAND, reward_amount)
    C->>C: UPDATE coupon -> PENDING
    Note over C: COMMIT → release lock → ACK
    C->>MQ: relay publish: redeem.command
    C->>C: tracking -> COMMAND_SENT
    MQ->>W: redeem.command.queue
    W->>W: INSERT wallet_transactions (idempotent)
    W->>W: INSERT outbox(REDEEM_RESULT)
    W->>MQ: relay publish: redeem.result
    MQ->>C: redeem.result.queue
    alt success
        C->>C: tracking -> SUCCESS
        C->>C: coupon -> REDEEMED
    else 4xx
        C->>C: tracking -> FAILED
        C->>C: coupon -> AVAILABLE
        C->>MQ: outbox(NOTIFICATION_EVENT)
    end
Recovery Job (mỗi 5 phút, lock 600s): re-publish các record INIT/COMMAND_SENT quá cũ, max 5 lần retry → BLOCKED
04-coupon-state.mmd

Coupon Status — State Machine

BLOCKED là terminal đối với automation (Recovery Job hết 5 lần retry, không xác nhận được kết quả với Wallet) nhưng không terminal về business — admin phải kiểm tra log Wallet rồi set tay về REDEEMED hoặc AVAILABLE. 
stateDiagram-v2
    [*] --> AVAILABLE

    AVAILABLE --> PENDING: matched, REDEEM_COMMAND queued
    PENDING --> REDEEMED: wallet success (200)
    PENDING --> AVAILABLE: wallet 4xx fail
    PENDING --> BLOCKED: 5x retry exhausted
    AVAILABLE --> EXPIRED: expired_at passed (daily job)

    BLOCKED --> REDEEMED: admin verify + resolve
    BLOCKED --> AVAILABLE: admin verify + resolve

    REDEEMED --> [*]
    EXPIRED --> [*]

    note right of BLOCKED
        Terminal cho automation,
        KHÔNG terminal cho business.
        Admin kiểm tra Wallet log
        truoc khi resolve thu cong.
    end note

    note right of PENDING
        "Dang bi lock", cho wallet
        xac nhan. TTL implicit qua
        Recovery Job 5 phut.
    end note
05-rabbitmq-topology.mmd

RabbitMQ Topology — Exchanges, Queues, DLQ

Mỗi queue chính có một retry/DLQ path riêng. order.retry.exchange dùng TTL 5s + Dead Letter Exchange để tự động requeue về order.queue khi Redis lock busy. Các DLQ còn lại dùng cho poison message / manual intervention.
